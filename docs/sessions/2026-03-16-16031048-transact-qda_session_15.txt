     Automatic Spatial Containment Relations for SW/A Maps

     Context

     Session 14 established Social Worlds/Arenas maps with formation rendering
     (ellipses/rectangles). The missing piece: when elements are spatially
     inside formations, or formations overlap, the system must automatically 
     derive and maintain containment relations as naming acts in the stack.
     Every move/resize updates the containment state вҖ” no manual trigger, no
     confirmation dialog. The spatial arrangement is the researcher's analytical
      judgment; the system takes it seriously.

     Design Decisions

     1. Fully automatic: containment computed and synced on every dragend and
     resizeend
     2. All changes are naming acts: containment relations created/removed via
     the existing relateElements() infrastructure, append-only in the stack
     3. Client-side geometry, server-side relations: math runs in browser (all
     positions already in memory), relation mutations go through the API
     4. AI agent behavior on containment changes: deferred to future session (AI
      is currently hyperactive, needs real-world calibration)
     5. No visualization of intersection areas: only detection and relation
     registration

     Geometry Module

     New file: src/lib/canvas/geometry.ts

     Three pure functions:

     pointInFormation(point, formation) вҶ’ boolean

     - Rotate point into formation-local coordinates (accounting for rotation)
     - Ellipse test: (localX/rx)ВІ + (localY/ry)ВІ вүӨ 1
     - Rectangle test (organization): |localX| вүӨ rx && |localY| вүӨ ry
     - Shape determined by SW_ROLE_STYLES[swRole].shape

     formationsOverlap(f1, f2) вҶ’ boolean

     - Sample 16 boundary points of f1, test each against f2 (and vice versa)
     - Also test center-in-center (catches full containment without boundary
     intersection)
     - Works for ellipse-ellipse, ellipse-rect, rect-rect combinations
     - Sufficient precision for typical 5-20 formations per map

     formationContainsFormation(inner, outer) вҶ’ boolean

     - All boundary sample points of inner lie inside outer
     - Distinguishes full containment from partial overlap

     computeSpatialRelations(elements, formations) вҶ’ SpatialState

     interface SpatialState {
       elementInFormation: Array<{ elementId: string; formationId: string }>;
       formationInFormation: Array<{ innerId: string; outerId: string }>;
       formationOverlaps: Array<{ formationA: string; formationB: string }>;
     }
     - Iterates all non-formation elements against all formations
     - Iterates all formations against each other:
       - If A fully inside B вҶ’ formationInFormation (directed: B contains A)
       - If partial overlap вҶ’ formationOverlaps (symmetric)
     - An element inside 2+ formations gets one entry per formation

     Spatial Relations

     Three relation types, all using the existing relation system with
     properties.spatiallyDerived: true:

     1. Containment: contains (directed)

     - valence: 'contains'
     - direction: directed_from = formation, directed_to = contained element (or
      inner formation)
     - symmetric: false
     - Element in formation, or formation fully inside another formation.

     2. Overlap: overlaps (symmetric)

     - valence: 'overlaps'
     - direction: none (symmetric)
     - symmetric: true
     - Two formations partially intersect (neither fully contains the other).

     3. Intersection positioning (Relationsrelationen)

     When an element is inside 2+ overlapping formations, it has contains
     relations to each. The intersection configuration вҖ” the
     dispositif-character of the position вҖ” is not a separate entity but emerges
      from the combination of:
     - ElementвҶ’FormationA (contains)
     - ElementвҶ’FormationB (contains)
     - FormationAвҶ”FormationB (overlaps)

     This triad is queryable вҖ” formallogisch trivial:
     - "Which elements lie in AвҲ©B?" = elements with contains to A AND contains
     to B, where AвҶ”B overlaps
     - "Which formations overlap at element X?" = formations that contain X and
     overlap each other
     - "Which elements are exclusively in A but not in any intersection?" =
     elements with contains to A only

     No additional data structure needed вҖ” pure set logic over existing
     relations. The AI can derive intersection positioning from the relation
     list alone.

     All relations: inscription empty (spatial fact is the act; researcher can
     inscribe later).

     Sync Flow

     After every handleNodeDragEnd and onresizeend (only on social-worlds maps):

     1. Compute: computeContainment() with current positions/dimensions
     2. Diff: Compare computed state against existing relations where
     properties.spatiallyDerived === true (both contains and overlaps)
     3. Sync: Single API call mapAction('syncSpatialRelations', { add: [...], 
     remove: [...] })
     4. Reload: refresh map data

     New API action: syncSpatialRelations

     Server handler in +server.ts:
     - add: array of { sourceId, targetId, valence, symmetric } вҖ” calls
     relateElements() for each with spatiallyDerived: true
     - remove: array of relation naming IDs вҖ” soft-delete via existing withdraw
     pattern
     - Handles both contains (directed) and overlaps (symmetric) in one call
     - Does NOT trigger AI agent (spatial sync is high-frequency; AI
     hyperactivity is a known issue)

     New query: withdrawRelation(namingId, mapId)

     Simple function in maps.ts that sets properties.withdrawn = true on the
     relation's appearance. Reuses existing withdrawal pattern.

     Integration with Existing System

     - List view: spatial relations appear naturally in the Relations section вҖ”
     contains (directed fromвҶ’to) and overlaps (symmetric). No special rendering
     needed.
     - Canvas connections: Filter out spatiallyDerived relations from

     - Cross-map: containment relations are map-specific (perspectival). Same
