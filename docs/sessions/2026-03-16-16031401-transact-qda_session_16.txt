 SW/A Analytical Support Structure                                                                                                                 
                                                        
 Context

 Sessions 14-15 built the rendering layer for Social Worlds/Arenas maps: formation types (ellipse/rect), resize/rotate, spatial relation
 derivation, Clarke demo project. The AI agent currently treats SW/A maps identically to SitMaps — same prompt, same tools, no formation
 awareness.

 The analytical support structure is what makes SW/A maps methodologically valuable. Clarke's 14 Social World questions and 11 Arena questions,
 the 5 analytical deepening moments, and cross-map integration give the researcher structured analytical scaffolding — not as a checklist, but as
 sensitizing prompts.

 Implementation Steps

 Step 1: Clarke questions + deepening moments as shared constants

 File: src/lib/shared/constants.ts

 Add three exported arrays:
 - CLARKE_SW_QUESTIONS: 14 questions for Social Worlds (work, sites, technologies, organizations, commitments, arenas, issues, discourses,
 identities, boundaries, segments, implicated actors, silences, intersections)
 - CLARKE_ARENA_QUESTIONS: 11 questions for Arenas (active worlds, contested issues, implicated actors, positions, discourses, organizations,
 history, power dynamics, hot issues, silences, boundary objects)
 - ANALYTICAL_DEEPENING: 5 moments as {key, label, question} objects (Stabilisierung, Konflikt, Dispositiv, Diskursive Konstitution, Cross-Map)

 These are imported by both prompts.ts (server-side, for AI) and +page.svelte (client-side, for stack panel).

 Step 2: SW/A system prompt supplement

 File: src/lib/server/ai/prompts.ts

 Add SWA_SUPPLEMENT constant (~60-80 lines). NOT a replacement for SYSTEM_PROMPT — appended when mapType === 'social-worlds'. Contains:
 - What formations ARE (universes of discourse, dispositif configurations)
 - Formation roles with visual signatures
 - Clarke's 14 SW questions (imported from constants, formatted as numbered list)
 - Clarke's 11 Arena questions (same)
 - 5 deepening moments with instructions for context-sensitive use
 - Spatial semantics (containment = membership, overlap = contested boundaries)
 - Key difference from SitMaps: "what is in the situation?" vs "how is it organized, stabilized, contested?"

 Add addFormation case to describeTrigger():
 Researcher created formation "${inscription}" (role: ${swRole})

 - spatialRelations?: Array<{type, formationA, formationB}> at context level

 In buildContextMessage(), when ctx.mapType === 'social-worlds':
 - Show [social-world], [arena], [discourse], [organization] role tag after designation for elements with swRole
 - Add SPATIAL STRUCTURE: section with containment/overlap info (skip if empty)
 - Add CROSS-MAP CONTEXT: section showing which SitMap elements connect to formations (skip if empty)

 Step 4: Cross-map participations query

 File: src/lib/server/db/queries/maps.ts

 New function getCrossMapParticipations(mapId: string, projectId: string):

 - LIMIT 30 to prevent context explosion
 - Only called for SW/A maps

 Memo System Redesign: From Passive Storage to Active Inquiry Medium

 Context

 The AI agent produces analytically valuable output (Clarke questions, provenance analysis, tensions, constellation observations) via write_memo —
  but these memos disappear into the project memo system, requiring manual navigation to find. The 4-second "AI: memo" toast is the only
 visibility. Meanwhile, AI structural cues (suggest_element, suggest_relation) have full lifecycle support: visible on map, discussable,
 withdrawable.

 This is not just a UX gap — it's an ontological inconsistency. The platform's transactional ontology (Dewey/Bentley) demands a coherent
 treatment.

 Ontological Foundation (D/B, Knowing and the Known, Ch. VI)

 "A name is in effect a truncated description. A description is an expanding naming." (§V)

 D/B treat Designation and Description as two aspects of the same process. But this applies to ONE naming — the description of "Beruf" is the
 expanding form of Beruf. Inquiry itself is not a naming — it is the medium in which namings are produced: "Specification is the type of naming
 that develops when inquiry gets down to close hard work."

 This yields two distinct memo modes:

 Mode A: Description-Memos (Stack-integral)

 - The expanding form of a single naming
 - Created during naming acts (rename, designate, relate)
 - Part of the naming's stack — part of what the naming IS
 - Already implemented as naming-act transparency mechanism
 - No changes needed

 Mode B: Analytical Memos (Inquiry-medium)

 - Reflexive inquiry process spanning constellations
 - NOT automatically a naming in the D/B sense
 - The medium of inquiry, not its product
 - Can be PROMOTED to a naming if the researcher explicitly decides so
 - AI memos are this type
 - Currently broken: invisible, non-dialogic, flat list

 Mode C: Organizational Memos (Process-medium)

 - Reflexion über den Forschungsprozess selbst: Erhebungsstrategie, Sampling-Entscheidungen, methodische Reflexionen
 - Bezugspunkt: nicht Daten/Konstellationen, sondern der Forschungsprozess
 - Arise from ongoing analysis ("theoretical sampling" in Grounded Theory / SA)
 - Not linked to specific namings but to the project or map as a whole
 - Example: "Based on the emerging arena structure, we should interview union representatives next"
 - Same lifecycle as Mode B (presented → discussed → acknowledged) but different scope

 Design Principle

 ▎ Analytical memos are the inquiry medium — visible where they matter, dialogic, but not automatically namings. They have their own lifecycle
 (presented → discussed → acknowledged/promoted), distinct from CCS designation (cue → characterization → specification).

 Memo System Redesign: From Passive Storage to Active Inquiry Medium                                                                               
                                                        
 Context

 The AI agent produces analytically valuable output (Clarke questions, provenance analysis, tensions, constellation observations) via write_memo —
  but these memos disappear into the project memo system, requiring manual navigation to find. The 4-second "AI: memo" toast is the only
 visibility. Meanwhile, AI structural cues (suggest_element, suggest_relation) have full lifecycle support: visible on map, discussable,
 withdrawable.

 This is not just a UX gap — it's an ontological inconsistency. The platform's transactional ontology (Dewey/Bentley) demands a coherent
 treatment.

 Ontological Foundation (D/B, Knowing and the Known, Ch. VI)

 "A name is in effect a truncated description. A description is an expanding naming." (§V)

 D/B treat Designation and Description as two aspects of the same process. But this applies to ONE naming — the description of "Beruf" is the
 expanding form of Beruf. Inquiry itself is not a naming — it is the medium in which namings are produced: "Specification is the type of naming
 that develops when inquiry gets down to close hard work."

 This yields two distinct memo modes:

 Mode A: Description-Memos (Stack-integral)

 - The expanding form of a single naming
 - Created during naming acts (rename, designate, relate)
 - Part of the naming's stack — part of what the naming IS
 - Already implemented as naming-act transparency mechanism
 - No changes needed

 Mode B: Analytical Memos (Inquiry-medium)

 - Reflexive inquiry process spanning constellations (data-level) or the research process itself (organizational: Erhebungsstrategie, Sampling,
 methodische Reflexionen)
 - Technically namings in the DB (stored in namings + memo_content) — but do not appear on any map perspective
 - The medium of inquiry, not its product
 - Can be PROMOTED to a map-visible naming if the researcher explicitly decides so
 - AI memos are this type
 - Currently broken: invisible, non-dialogic, flat list

 Design Principle

 ▎ Analytical memos are the shared inquiry medium of the Interpretations-Kollektiv (researcher + AI as co-inquirers). They are visible where they
 matter, dialogic, and author-agnostic in structure. Provenance (who authored, in what context, with what epistemic authority) is a transparent
 property of every memo — not a type distinction that determines different UI or lifecycle. AI-Kritik is embedded as provenance awareness, not as
 ontological separation.

 This means:
 - No "AI memo" vs "human memo" as separate categories
 - ONE memo system, ONE discussion mechanism, ONE visibility pattern
 - Every memo carries provenance: author, trigger context, epistemic grounding
 - AI authorship is a provenance marker that signals "needs critical evaluation" — not a different kind of memo
 - The researcher can write analytical memos in the same medium, and the AI can respond
 - The AI can be invited to discuss researcher memos, and the researcher discusses AI memos — symmetrical

 Context

 Research data portability is a core requirement. Currently all data lives in a single PostgreSQL database. The target:
  each project = one .db file. Export = copy file. Import = place file. Sharing = send one file. This also eliminates
 the complex duplicateProject remap logic (becomes cp).

 Stages 1+2 of the memo system are complete. Migrating the DB layer now means all future work (memo Stage 3/4, new map
 types) builds on the final architecture.

 Architecture

 data/
 ├── central.db              ← users, sessions, projects, project_members
 └── projects/
     ├── {project-id-1}.db   ← all project-scoped tables
     ├── {project-id-2}.db
     └── ...

 - Central DB (always open): users, sessions, projects, project_members
 - Project DB (opened on demand, cached, WAL mode): namings, participations, appearances, naming_acts,
 phase_memberships, topology_snapshots, document_content, memo_content, researcher_namings, ai_namings, ai_interactions
 - Driver: better-sqlite3 (synchronous, fast, JSON1 built-in)

 Design Decisions

 1. Drop project_id from project-scoped tables — the file IS the project. Queries lose WHERE n.project_id = $1,
 parameter indices shift.
 2. Drop FK constraints to users in project DBs (can't cross-DB enforce). Keep user_id/created_by column values.
 3. Cross-DB queries (e.g. namings.ts fetches users.display_name): split into two queries.
 4. seq columns: Triggers for namings.seq. For naming_acts.seq (per-naming), use SELECT COALESCE(MAX(seq),0)+1 in app.
 5. Wrapper API shape: Return { rows: T[] } matching current pg shape to minimize downstream changes.
 6. Synchronous is fine: Single-user research tool. WAL mode for concurrent AI agent reads.
