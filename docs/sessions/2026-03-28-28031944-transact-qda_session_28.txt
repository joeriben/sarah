  ---
  Diagnose

  Es gibt zwei Scroll-Container übereinander, die sich gegenseitig bekämpfen:

  1. .project-content hat overflow-y: auto — das ist der Scroll-Container der
  gesamten Projekt-Seite
  2. .doc-body hat height: calc(100vh - 7rem) — das ist ein harter
  Viewport-Wert, der die Parent-Kette ignoriert

  Das Ergebnis: .doc-body ist entweder zu groß (überläuft .project-content →
  Doppelscroll) oder zu klein (verschenkt Platz). Der calc(100vh - 7rem)
  berücksichtigt weder die padding: 2rem von .project-content noch den
  .doc-header innerhalb der Page. Es ist ein Schuss ins Dunkle.

  Außerdem: .code-margin sitzt innerhalb von .content-panel (innerhalb von
  .text-with-margin). Das heißt: sie scrollt mit dem Dokument mit. Das ist der
  Grund, warum die Margin-Labels nie fixiert waren.

  ---
  Zielzustand

  Verhalten

  │       Element        │        Scrollt?         │      Bleibt fest?      │
  ├──────────────────────┼─────────────────────────┼────────────────────────┤

  ├──────────────────────┼─────────────────────────┼────────────────────────┤

  Architektur

  Die Kette muss lückenlos höhenbegrenzt sein. Kein calc(100vh - ...), sondern
  reines Flex-Flow:

  .app              flex column, height: 100vh (nicht min-height!)
    .app-header     flex-shrink: 0
    .content        flex: 1, min-height: 0, overflow: hidden (NICHT auto!)
      .project-layout   flex row, height: 100%
        .project-sidebar  width: 200px, flex-shrink: 0
        .project-content  flex: 1, min-height: 0, overflow: hidden (NICHT
  auto!), padding: 2rem
          .doc-viewer     display: flex, flex-direction: column, height: 100%,
  min-height: 0
            .doc-header   flex-shrink: 0
            .doc-body     flex: 1, display: flex, gap: 1rem, min-height: 0,
  overflow: hidden
              .content-panel   flex: 1, min-height: 0, overflow-y: auto  ←
  EINZIGER Scroll
                .document-text   (pre, fließt normal)
              .code-margin     width: 100px, flex-shrink: 0, position: relative,
   overflow: hidden
              .work-panel      width: 280px, flex-shrink: 0, overflow-y: auto
  (eigener Scroll)

  Schlüsselentscheidungen

  1. .content und .project-content verlieren overflow-y: auto → werden overflow:
   hidden. Es gibt genau einen Scroll-Container: .content-panel. Damit scrollt
  die Sidebar nie, das Work-Panel nie, die Code-Margin nie.
  2. .code-margin wird aus .text-with-margin herausgelöst und zum direkten Kind
  von .doc-body. Begründung: Wenn sie innerhalb von .content-panel bleibt,
  scrollt sie mit dem Dokument. Als Flex-Sibling von .content-panel bleibt sie
  stehen. Die Margin-Labels sind bereits position: absolute mit JS-berechneten
  top-Werten — diese müssen dann relativ zum sichtbaren Viewport der
  .content-panel berechnet werden (unter Abzug von scrollTop), nicht relativ zum
   Dokument-Textflow.
  3. calc(100vh - 7rem) fällt weg. Die Höhe ergibt sich rein aus der Flex-Kette:
   100vh → minus Header → minus Sidebar-Padding → fertig. Kein manueller
  Rechenwert nötig.
  4. .doc-viewer bekommt height: 100% und min-height: 0 — damit füllt es
  .project-content exakt aus und erlaubt dem Flex-Kind .doc-body, den restlichen
   Raum zu nutzen.

  Code-Margin Label-Positionierung

  Da die Code-Margin jetzt nicht mehr mit dem Dokument scrollt, müssen die
  Labels dynamisch repositioniert werden:

  - Aktuell: label.top = annotationSpan.offsetTop (relativ zum Dokument)
  - Neu: label.top = annotationSpan.getBoundingClientRect().top - 
  codeMargin.getBoundingClientRect().top
  - Das heißt: Labels werden relativ zum sichtbaren Bereich positioniert
  - Labels außerhalb des sichtbaren Bereichs bekommen display: none
  - Das Recalc muss auf dem scroll-Event von .content-panel feuern (debounced)

  Was sich NICHT ändert

  - .annotations-overlay bleibt position: fixed — funktioniert unabhängig
  - .margin-tooltip bleibt position: fixed — ok
  - Work-Panel-Inhalt bleibt gleich
  - Alle funktionalen Features aus Session 28 bleiben erhalten

  5. Annotations-Overlay bleibt funktional und unabhängig
  6. Langes Dokument laden → kein Doppelscrollbar, kein Overflow irgendwo anders

     Layout-Fix: Browser-Scrollbar, Sticky Panels

     Context

     Session 28 hat 5 gescheiterte Layout-Versuche produziert, weil die
     Parent-Kette nicht verstanden und mit calc(100vh), position: fixed, und
     verschachtelten Scroll-Containern gearbeitet wurde. Die korrekte Lösung ist
      normales Webdesign: der Browser scrollt, alles Fixe ist position: sticky.

     Zielzustand

     - Eine Scrollbar: die des Browsers (bzw. .content als äußerster
     Scroll-Container mit overflow-y: auto; height: 0; flex: 1)
     - Sticky: Header, Sidebar, Work-Panel, Code-Margin
     - Einzige Inline-Scrollbar: Annotations-Overlay
     - Kein calc(100vh - ...) auf Layout-Elementen
     - Kein position: fixed auf Layout-Elementen (nur Overlay + Tooltips)

     Änderungen
