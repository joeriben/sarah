  "There is no such thing as context" (Clarke) means discourses, theories, policy documents are IN  
  the situation, not external frames applied to it. So a Foucault chapter, a policy brief, a media
  article — these are data, not context. They belong in the corpus as uploadable, codeable          
  documents.                                                

  This collapses the provenance question cleanly:                                                   
  
  Grounding = document anchor. Period. Whether the document is an interview transcript, a           
  photograph, a policy text, or a chapter from Foucault — it's in the corpus, it's codeable, and a
  naming grounded in it gets 📄.                                                                    
                                                            
  Memos (📝) are a different dimension entirely — they're the researcher's reflexive process, not a 
  grounding mechanism. A memo documents why you named something, not what grounds it. So 📝 and 📄
  are orthogonal:                                                                                   
                                                            
  ┌──────────────────────────┬──────────────────────────┬──────────────────────────────────────┐    
  │                          │         No memo          │            Has memo (📝)             │
  ├──────────────────────────┼──────────────────────────┼──────────────────────────────────────┤    
  │ No document anchor       │ ∅ ungrounded             │ Reflected but ungrounded — flag this │
  ├──────────────────────────┼──────────────────────────┼──────────────────────────────────────┤
  │ Has document anchor (📄) │ Grounded but unreflected │ Fully accountable                    │    
  └──────────────────────────┴──────────────────────────┴──────────────────────────────────────┘    
                                                                                                    
  The ∅ case is legitimate early in analysis (fresh cues), but should be resolved. The 📝-without-📄

 Code System Refactor: Codes as Derived View from Maps                                              
                                                                                                  
 Context

 The code system currently uses a separate code-system perspective — codes are pre-created
 categories applied to data (ATLAS.ti model). This is inconsistent with Situational Analysis where
 codes emerge from the mapping process. Per docs/design-provenance-and-codes.md, the maps are the
 analytical center, and the code list should be a derived view of namings used as annotation
 anchors.

 Goal: The annotation sidebar draws from map elements. The "Codes" page becomes a grounding
 dashboard. In-vivo coding still works.

 Key Design Decision: Grounding Workspace

 Namings have NO properties column — properties only exist on appearances. In-vivo codes (created
 during annotation without a map context) need an appearance somewhere for their color.

 Solution: Rename code-system perspective to grounding-workspace. It serves as a holding pen for
 orphan in-vivo codes not yet placed on a map. It also continues to host annotation relation
 appearances (valence: 'codes'). This is NOT a separate analytical domain — it's infrastructure.

 Changes

 Step 1: Migration — migrations/008_retire_code_system.sql

 Rename the role on existing code-system perspectives:

 UPDATE appearances
 SET properties = jsonb_set(properties, '{role}', '"grounding-workspace"')
 WHERE naming_id = perspective_id
   AND mode = 'perspective'
   AND properties->>'role' = 'code-system';

 No data deleted. All existing codes and annotations preserved.

 'code-system' to role: 'grounding-workspace')

 workspace

 - createAnnotation() — no change (already uses grounding workspace perspective for the relation
