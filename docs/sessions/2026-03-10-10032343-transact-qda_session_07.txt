● Gute Recherche-Ergebnisse. Hier die Zusammenfassung:

  Status quo in transact-qda

  Ihr habt bereits alles Wesentliche vorbereitet:
  - document_content Tabelle mit full_text, file_path, mime_type
  - createAnnotation() mit anchorType: 'text' | 'image_region' und anchor: { pos0, pos1 }
  - Upload-API mit Text-Extraktion (PDF via pdf-parse, DOCX via mammoth)
  - Annotations-API (GET/POST/DELETE) fertig
  - Document Viewer existiert als Split-View (Text links, Codes rechts) — aber noch ohne interaktive
   Annotation

  Text-Annotation: CSS Custom Highlight API

  Die klare Empfehlung — keine Library nötig:

  1. User selektiert Text → window.getSelection() → pos0/pos1 berechnen
  2. Speichern via existierendem createAnnotation() mit anchor: { pos0, pos1 }
  3. Rendering via CSS.highlights.set() + ::highlight(code-xxx) — kein DOM-Mutation, rein visuelles
  Overlay
  4. Dark-Theme-Styling trivial, Svelte 5 kompatibel, zero Dependencies

  Alle QDA-Tools (QualCoder, Taguette, ATLAS.ti) speichern intern genauso: Character-Offset-Paare.
  Eure Datenstruktur ist bereits korrekt.

  Zu Collabora/Nextcloud

  Ja, für Dokument-Rendering (DOCX, PDF layout-getreu anzeigen) gibt es fertige Lösungen. Aber die
  Frage ist: braucht ihr das?

  - Collabora Online = LibreOffice im Browser. Voll-Editor, WOPI-Protokoll, Docker-Container. Schwer
   für Annotation-Overlay zu instrumentieren.
  - Nextcloud = File-Management-Plattform, nicht Annotation-Tool.

  Für QDA-Codierung ist layout-getreues Rendering irrelevant. Was zählt, ist der Text-Inhalt — und
  den habt ihr bereits extrahiert. Taguette, QualCoder, ATLAS.ti — alle zeigen zum Codieren den
  Rohtext (oder simplifiziertes HTML), nicht das Original-Layout. Der Grund: Character-Offsets
  müssen stabil sein, und das geht nur auf einer kanonischen Textrepräsentation.

  Mein Vorschlag:

  ┌───────────────┬─────────────────────────────────────┬──────────────────────────────────────┐
  │ Dokument-Typ  │        Anzeige zum Codieren         │           Original-Ansicht           │
  ├───────────────┼─────────────────────────────────────┼──────────────────────────────────────┤
  │ Text/Markdown │ full_text mit Highlight-API         │ —                                    │
  ├───────────────┼─────────────────────────────────────┼──────────────────────────────────────┤
  │ PDF           │ Extrahierter Text mit Highlight-API │ PDF-Viewer (embed/iframe) als Toggle │
  ├───────────────┼─────────────────────────────────────┼──────────────────────────────────────┤
  │ DOCX          │ Extrahierter Text mit Highlight-API │ Optional: mammoth → HTML             │
  ├───────────────┼─────────────────────────────────────┼──────────────────────────────────────┤
  │ Bild          │ Annotorious (vanilla JS)            │ Direkt <img>                         │
  └───────────────┴─────────────────────────────────────┴──────────────────────────────────────┘

  Die "Original-Ansicht" wäre ein Nice-to-have-Toggle, aber zum Codieren arbeitet man immer auf dem
  extrahierten Text.

  R-Integration?

  Nicht sinnvoll. RQDA ist abandoned, qcoder minimal. Die analytischen Features (Wort-Frequenz,
  Co-Occurrence, Inter-Coder-Reliability) sind direkt in SQL/TypeScript besser aufgehoben. Eure
  transaktionale Ontologie ist bereits mächtiger als alles, was R-QDA-Pakete modellieren.

  ---
  Soll ich mit der Text-Annotation-UI anfangen? Das wäre der Document Viewer
  (documents/[docId]/+page.svelte) mit Selection-Handler und CSS Highlight API — der größte fehlende
   Baustein für situationales Codieren.

 Image Analysis and Coding

 Context

 Text annotation is now working (CSS Custom Highlight API, pos0/pos1 anchors, sidebar code picker).
  The next step is image coding — selecting regions of images and applying codes.

 Methodological Landscape

 Clarke & Situational Analysis: Clarke discusses visual materials primarily as discourse — images
 are discursive elements in the situation, analyzed via visual discourse analysis (drawing on Rose,
  Sturken & Cartwright). In SA, you can make a situational map of an image (what human/nonhuman
 elements, discourses, relations are present in/constituted by the image). But Clarke doesn't
 prescribe a specific visual analysis technique — she provides the mapping framework, not the
 image-reading method. So the tool needs to support multiple visual analysis traditions within SA's
  mapping framework.

 What the different approaches need from the tool:

 D/B Gradient and Visual Analysis

 The designation process (cue ↔ characterization ↔ specification) maps differently onto visual
 analysis than onto text coding — and differently across traditions:

 The region-marking act itself is a cue. Selecting a rectangle on an image is analogous to
 highlighting text — it's a pointing gesture, a "something here." It sits at the D-pole
 (Andeutung). The code application moves toward characterization. The comment toward specification.
  This gradient already exists implicitly in the text annotation system, but for images the
 methodological traditions make it explicit:

 ┌───────────┬────────────────────────────────────┬────────────────────────────────────────────┐
 │ Tradition │     D-pole (cue/indeterminacy)     │    B-pole (specification/determination)    │
 ├───────────┼────────────────────────────────────┼────────────────────────────────────────────┤
 │ Panofsky  │ Pre-iconographic: "I see a raised  │ Iconological: "This signifies cultural     │
 │           │ hand"                              │ authority of..."                           │
 ├───────────┼────────────────────────────────────┼────────────────────────────────────────────┤
 │           │ Sehendes Sehen — iconic seeing     │                                            │
 │ Imdahl    │ resists specification. Composition │ Wiedererkennendes Sehen —                  │
 │           │  lines are tools for maintaining   │ recognizing/classifying what is depicted   │
 │           │ indeterminacy, not coding.         │                                            │
 ├───────────┼────────────────────────────────────┼────────────────────────────────────────────┤
 │           │                                    │ Reflecting interpretation (Wie) — but this │
 │ Bohnsack  │ Formulating interpretation (Was)   │  is not "more specified", it's a different │
 │           │                                    │  dimension (modus operandi/habitus)        │
 ├───────────┼────────────────────────────────────┼────────────────────────────────────────────┤
 │ Breckner  │ Each segment analysis opens        │ Cross-segment synthesis moves toward       │
 │           │ meaning — deliberately anti-B      │ characterization                           │
 ├───────────┼────────────────────────────────────┼────────────────────────────────────────────┤
 │ Serial    │ Individual image = cue             │ Cross-image pattern = specification        │
 │           │                                    │ through comparison                         │
 └───────────┴────────────────────────────────────┴────────────────────────────────────────────┘

 Key implication for the tool: Imdahl's composition drawings (planimetric lines, perspective axes)
 are epistemologically different from Panofsky's region-coding. They are not annotations moving
 toward B — they are instruments for staying at D, for "seeing seeing." If we later build
 composition tools, they should NOT be stored as valence: 'codes' annotations but as a separate
 layer (perhaps valence: 'composition'), because they serve a fundamentally different
 epistemological function.

 For Phase 1 (rectangle region coding), the D/B gradient is handled naturally:
 - The region selection = cue (pointing gesture)
 - Code application = characterization (naming what's there)
 - Comment = further specification
 - The same region can receive multiple annotations at different D/B levels (multiple codes, each a
  different designation in the chain)
 - Different perspectives/maps can collapse differently — one analyst's Panofsky Level 1 reading,
 another's Level 3

 Clarke's SA as meta-framework: Clarke's situational mapping operates on a different level than the
  image-internal analysis methods. SA maps the situation (what elements, discourses, relations does
  the image participate in?), while Panofsky/Imdahl/Bohnsack analyze the image internally. The
 platform supports both: image-internal annotations (region coding) feed into map-level elements (a
  code grounded in image evidence shows 📄 provenance on the canvas).

 Design principle: All approaches share the basic operation of marking and annotating image
 regions. The specialized tools (composition lines, segmentation, serial comparison) are extensions
  with distinct epistemological status. Phase 1 builds the foundation — region coding as

│ D/B Gradient and Visual Analysis                                                                 │
│                                                                                                  │
│ The designation process (cue ↔ characterization ↔ specification) maps differently onto visual    │
│ analysis than onto text coding — and differently across traditions:                              │
│                                                                                                  │
│ The region-marking act itself is a cue. Selecting a rectangle on an image is analogous to        │
│ highlighting text — it's a pointing gesture, a "something here." It sits at the D-pole           │
│ (Andeutung). The code application moves toward characterization. The comment toward              │
│ specification. This gradient already exists implicitly in the text annotation system, but for    │
│ images the methodological traditions make it explicit:                                           │
│                                                                                                  │
│ ┌───────────┬────────────────────────────────────┬────────────────────────────────────────────┐  │
│ │ Tradition │     D-pole (cue/indeterminacy)     │    B-pole (specification/determination)    │  │
│ ├───────────┼────────────────────────────────────┼────────────────────────────────────────────┤  │
│ │ Panofsky  │ Pre-iconographic: "I see a raised  │ Iconological: "This signifies cultural     │  │
│ │           │ hand"                              │ authority of..."                           │  │

│ │           │ Sehendes Sehen — iconic seeing     │                                            │  │

│ │           │  lines are tools for maintaining   │ recognizing/classifying what is depicted   │  │
│ │           │ indeterminacy, not coding.         │                                            │  │
│ ├───────────┼────────────────────────────────────┼────────────────────────────────────────────┤  │
│ │           │                                    │ Reflecting interpretation (Wie) — but this │  │
│ │ Bohnsack  │ Formulating interpretation (Was)   │  is not "more specified", it's a different │  │

│ ├───────────┼────────────────────────────────────┼────────────────────────────────────────────┤  │
│ │ Breckner  │ Each segment analysis opens        │ Cross-segment synthesis moves toward       │  │
