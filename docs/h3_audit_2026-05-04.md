# H3 Audit 2026-05-04 — Stuck-Guard-Wurzelfix-Vorlauf

**Status:** Read-only Befund. Kein Code geändert. Lösungs-Diskussion separat.
**Auftraggeber:** User-Setzung am 2026-05-04 nach Handover Session 2.
**Verdacht:** "Ein Teil der H3-Logik wurde inkonsistent implementiert."
**Prämisse abgelehnt:** Session 1/2-Hypothese "Vite-SSR-Modul-Cache" — siehe §1.

Das Audit deckt drei Dinge auf:

1. **Stuck-Guard-Wurzel ist ein Atom-ID-Kollisions-Bug im Orchestrator**, nicht ein Modul-Lade-Problem.
2. **Massive Inkonsistenz** der Vorbedingungs- und Idempotenz-Behandlung über die 13 H3-Module hinweg.
3. **DB-Empirie auf Reproduktion-Case 6bc7208c** bestätigt den Bug: Run scheitert deterministisch nach exakt 2 erfolgreichen H3-Phasen, vor der dritten.

---

## §1 Refutation: Vite-SSR-Modul-Cache war nicht die Wurzel

Das Handover Session 2 (`docs/h3_handover_2026-05-04_session2.md`) hatte als Hypothese:

> "Stuck-Guard greift, weil Vite den vorherigen Modul-Stand cached und der gefixte Code nicht in den Worker-Prozess gelangt."

Diese Hypothese ist nicht haltbar. Belege:

- **Statische Imports.** `src/lib/server/pipeline/h3-phases.ts:23-43` importiert alle 9 H3-Module statisch. Keine dynamischen `import()` zur Laufzeit. Vite-SSR cached statische Imports auf File-mtime-Basis; jede Edit invalidiert den Cache deterministisch.
- **Reload-Resilience.** Der Stuck-Guard lebt im Orchestrator (`src/lib/server/pipeline/orchestrator.ts:798-815`), nicht in den H3-Modulen. Selbst wenn ein H3-Modul gecacht wäre, würde der Stuck-Guard-Vergleich `next.atom.id === lastProcessedAtomId` davon nicht beeinflusst.
- **DB-Persistenz von Konstrukten** (siehe §4) zeigt, dass die H3-Module ihren aktuellen Code-Stand ausführen — sie persistieren Konstrukte erfolgreich. Wäre der Cache schuld, hätte ein alter Code-Stand entweder andere Konstrukte produziert oder gar keine.

Die Modul-Cache-Hypothese hat ein realeres Symptom (langsamer dev-Loop bei großen Edits) zu einer falschen Diagnose verdichtet.

## §2 Wurzelbefund: Atom-ID-Kollision im Stuck-Guard

### §2.1 Mechanik

Der Orchestrator hat eine zentrale Schleife, die Phasen in `phasesForRun(options)`-Reihenfolge abarbeitet (`orchestrator.ts:737-872`). In jedem Schleifendurchlauf:

1. **Atom-Suche** (Zeilen 749-767): `listAtomsForPhase(phase, documentId)` für jede Phase, bis eine Phase mit `realPending.length > 0` gefunden ist.
2. **Stuck-Guard** (Zeilen 798-815):
   ```ts
   if (next.atom.id === lastProcessedAtomId) {
     sameAtomRepeatCount += 1;
     if (sameAtomRepeatCount >= 3) {
       // markFailed; return
     }
   } else {
     lastProcessedAtomId = next.atom.id;
     sameAtomRepeatCount = 1;
   }
   ```
3. **Pass-Ausführung** + Persistenz, danach nächster Schleifen-Durchlauf.

Der Vergleich ist **phasen-blind**: Nur `atom.id` wird verglichen, die Phase nicht. `lastProcessedAtomId` wird **niemals zurückgesetzt** beim Phasen-Wechsel oder nach erfolgreichem Pass — nur im `catch`-Pfad bei generischem Error (Zeilen 869-870).

### §2.2 H3-Atom-Modell

`listAtomsForPhase` (`orchestrator.ts:484-504`) gibt für **alle 9 H3-Phasen identisch** denselben `documentId` als `atom.id` zurück:

```ts
case 'h3_exposition':
case 'h3_grundlagentheorie':
case 'h3_forschungsdesign':
case 'h3_durchfuehrung':
case 'h3_synthese':
case 'h3_schlussreflexion':
case 'h3_exkurs':
case 'h3_werk_deskription':
case 'h3_werk_gutacht': {
  const all: AtomRef[] = [{ id: documentId, label: PHASE_LABEL[phase] }];
  ...
}
```

Konsequenz: Die 9 H3-Phasen sind aus Sicht des Stuck-Guards **derselbe Atom**, weil sie alle `documentId` als ID tragen. Das verletzt direkt die Memory-Setzung *"Atom ist AI-slop — pro Phase die echte Einheit benennen"*.

### §2.3 Kollisions-Sequenz

Beispiel-Trace bei einem Run mit `h3_exposition` → `h3_grundlagentheorie` → `h3_forschungsdesign`:

| Iter | Phase | atom.id | last…Id (vor) | counter (vor → nach) | Wirkung |
|---|---|---|---|---|---|
| 1 | h3_exposition | `docX` | `null` | `0 → 1` | else-Zweig: last=docX, counter=1 |
| 2 | h3_grundlagentheorie | `docX` | `docX` | `1 → 2` | match: counter=2 |
| 3 | h3_forschungsdesign | `docX` | `docX` | `2 → 3` | **STUCK → markFailed** |

Der Stuck-Guard feuert deterministisch beim **dritten** H3-Phasen-Wechsel, **bevor** der dritte Pass überhaupt ausgeführt wird (Stuck-Guard liegt vor `executeStep`, Zeilen 793-817). Die Fehlermeldung suggeriert "pass returned successfully 3× but listAtomsForPhase still marks it pending" — was sachlich **falsch** ist: Es waren 2 verschiedene Phasen erfolgreich, die dritte hat noch nie ausgeführt.

### §2.4 Warum H1 nicht betroffen ist

Bei H1-Phasen (z.B. `argumentation_graph`) gibt `listParagraphAtoms` pro Paragraph eine eigene `atom.id = paragraphId` zurück. Der Stuck-Guard-Vergleich differenziert dort korrekt zwischen Atomen, weil pro Paragraph unterschiedliche IDs vorliegen. Der Bug manifestiert sich nur in der H3-Schicht.

## §3 Inkonsistenz-Matrix der H3-Module

13 Module/Sub-Pässe. Drei orthogonale Achsen: Vorbedingungs-Fehler-Form, Idempotenz, Empty-Path.

**Konsequenz im Orchestrator-Catch** (`orchestrator.ts:841-871`):

- `PreconditionFailedError` → `markFailed`, sauberer Run-Stop, Reviewer-Eingriff.
- Generic `Error` → fail-tolerant: Atom wird in `erroredAtomIds` aufgenommen, Loop läuft mit nächstem Atom weiter.

Bei einer fehlenden Vorbedingung (z.B. fehlender FRAGESTELLUNG) verhalten sich die Heuristiken also **nicht einheitlich**:

- `forschungsdesign.ts` (FRAGESTELLUNG fehlt) → markFailed, Reviewer wird informiert ✓
- `schlussreflexion.ts` (FRAGESTELLUNG fehlt, Z. 709) → fail-tolerant geskippt, Run läuft "weiter" ✗
- `synthese.ts` (FRAGESTELLUNG fehlt, Z. 607) → fail-tolerant geskippt ✗

Das ist eine echte funktionale Inkonsistenz, nicht nur Code-Stil. Zwei Heuristiken (synthese, schlussreflexion) verletzen ihre eigene Vorbedingungs-Spec, weil sie für dieselbe Klasse Fehler beide Error-Typen verwenden.

### §3.3 Empty-Path-Verhalten (kein Material)

Wenn die Heuristik gestartet wird, aber das Material leer ist (z.B. keine GRUNDLAGENTHEORIE-Container, keine EXKURS-Container):

| Modul | Empty-Path |
|---|---|
| `exposition.ts` | `found=false` → reguläres Result mit `null`-IDs (kein Skip-Marker im StepResult) |
| `forschungsdesign.ts` | `PreconditionFailedError` (Reviewer-Eingriff erforderlich) |
| `durchfuehrung.ts` Step 1 | leeres Result, `tokens=0`, kein Fehler |
| `durchfuehrung.ts` Step 2 | Passthrough, `tokens=0` |
| `exkurs.ts` | No-op-Return wenn keine EXKURS-Container, `tokens=0` (keine Vorbedingungs-Prüfung) |
| `synthese.ts` | `PreconditionFailedError` für leere SYNTHESE-Container |
| `schlussreflexion.ts` | `PreconditionFailedError` für fehlendes SR-Material (mit "letztes Drittel"-Recovery) |
| `werk-deskription.ts` | `constructs.length===0` → nur Warning, fortfahren |
| `werk-gutacht.ts` | siehe `werk-gutacht.ts:556-586` — vier `PreconditionFailedError` für fehlende Voraussetzungen |

**Stand:** Drei Spielarten von Empty-Path-Verhalten:

- **(a) Reguläres Result** mit leerem Inhalt (`exposition`, `durchfuehrung-Step1`, `exkurs`, `werk-deskription`)
- **(b) PreconditionFailedError** mit Reviewer-Hint (`forschungsdesign`, `synthese`, `schlussreflexion`, `werk-gutacht`)
- Kein Modul gibt **`{ skipped: true }`** zurück, obwohl das StepResult-Schema das vorsieht.

Variante (a) ist gegen den Sinn von `skipped: true`: Der Pass läuft erfolgreich durch, der Done-Check (`isH3PhaseDoneForDocument`) findet aber **keine** Konstrukte, weil keine persistiert wurden. Beim nächsten Loop-Durchgang ist `pending` immer noch nicht-leer → der Stuck-Guard fängt das (was korrektes Verhalten wäre, **wenn** der Stuck-Guard nicht schon vorher feuern würde, siehe §2).

### §3.4 Token-Aggregation

`grundlagentheorie.ts` Step 1 (Verweisprofil): Der Wrapper in `h3-phases.ts:271-280` summiert **keine** Tokens für Step 1, mit Kommentar:

> "Daten-Verlust akzeptiert; das Genaue Tracking lebt im jeweiligen CLI-Skript-Output."

Konsequenz: Pipeline-Run-Token-Statistik ist für GTH systematisch zu niedrig. Nicht kritisch für das aktuelle Stuck-Guard-Problem, aber dokumentiert eine weitere Inkonsistenz, die bei zukünftigen Cost-Reports relevant wird.

## §5 Was nicht zum Wurzelfix gehört, aber gefunden wurde

Diese Befunde sind **sekundär** zum Stuck-Guard-Problem und werden hier nur erwähnt, damit sie nicht im selben Atemzug "mitgefixt" werden:

1. **EXPOSITION ohne clearExisting** (§3.2) — Re-Run produziert Duplikate.
2. **GTH-Pyramide ohne Idempotenz** (§3.2, 5 Sub-Pässe).
3. **EXKURS modifiziert FG-Stack ohne Re-Run-Schutz** — ein Re-Run von h3_exkurs würde re_spec doppelt anhängen.
4. **Inkonsistente Vorbedingungs-Errors in synthese + schlussreflexion** (§3.1) — fehlende FRAGESTELLUNG wirft generic Error statt PreconditionFailedError, wird also fail-tolerant verschluckt.
5. **GTH Step 1 Tokens werden nicht aggregiert** (§3.4).

Diese sind eigene Themen für eigene Diskussionen.

## §6 Zusammenfassung in einem Satz

Der Stuck-Guard markiert einen Run nach exakt zwei erfolgreichen H3-Phasen-Wechseln als "stuck", **weil alle 9 H3-Phasen denselben Atom-Namen `documentId` tragen und der Guard den Phasen-Wechsel nicht von einem fehlgeschlagenen Re-Pass desselben Atoms unterscheiden kann.** Modul-Cache, Heuristik-Logik und Persistenz-Layer sind **nicht** Teil der Wurzel.

## §7 Mögliche Lösungswege (zur Diskussion mit User, NICHT implementiert)

Drei orthogonale Achsen, an denen man drehen könnte. Jeder hat Implikationen für die Memory-Setzung *"Atom ist AI-slop — pro Phase die echte Einheit"*:

### Achse A — Atom-Identität phasen-eindeutig machen
- A1: H3-Atom-IDs als `${phase}:${documentId}` zusammensetzen.
- A2: Stuck-Guard auf `(atom.id, phase)`-Tupel vergleichen statt nur `atom.id`.

A2 ist der minimal-invasive Fix; A1 ändert das Atom-Modell global.

### Achse B — Stuck-Guard reset bei Phasen-Wechsel
- B: `if (next.phase !== lastPhase) { lastProcessedAtomId = null; sameAtomRepeatCount = 0; }` vor dem aktuellen Vergleich.

Behebt das aktuelle Symptom, hinterlässt aber den konzeptuellen Bruch, dass H3-"Atome" alle dieselbe ID tragen.

### Achse C — Atom-Modell auflösen (User-Memory: "atom is AI-slop")
- C: H3 nicht über die Atom-Schleife laufen lassen, sondern als eigene "Werk-Phase" mit Phasen-spezifischer Done-Logik. Pro Phase wird genau einmal `runH3Phase` aufgerufen, kein Atom-Iterator-Wrapper.

Das ist die radikalste Lösung und die einzige, die der Memory-Setzung *"Stuck-Guard ist Symptom, nicht Lösung"* treu bleibt — der Stuck-Guard wäre für H3 nicht mehr notwendig.

**Audit-Empfehlung:** Vor Code-Änderung User-Setzung zu Achse einholen. Die drei Achsen sind nicht orthogonal-additiv: A oder B macht C teils überflüssig und umgekehrt.
