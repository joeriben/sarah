## Was als Nächstes dran ist

### (b) Container-Orchestrator — empfohlener nächster Schritt

Heute laufen alle H3-Heuristiken als separate CLI-Skripte. Nicht-funktional bei realem Use: User will einen Pipeline-Run, nicht 7 Skripte hintereinander triggern.

**Aufgabe**: ein `runH3Pipeline(caseId, options)` in `src/lib/server/ai/h3/orchestrator.ts` (oder `src/lib/server/pipeline/h3-orchestrator.ts`), das die H3-Pässe in der korrekten Cross-Typ-Reihenfolge ausführt:

```
EXPOSITION
  → GRUNDLAGENTHEORIE Step 1 (VERWEIS_PROFIL)
  → GRUNDLAGENTHEORIE Step 2 (BLOCK_ROUTING)
  → GRUNDLAGENTHEORIE Step 3a (reproductive: H2 + ECKPUNKT_CHECK)
  → GRUNDLAGENTHEORIE Step 3b (discursive: DISKURSIV_BEZUG_PRÜFEN)
  → GRUNDLAGENTHEORIE Step 4 (FORSCHUNGSGEGENSTAND_REKONSTRUIEREN)
  → EXKURS (modifiziert FG via version_stack)
  → FORSCHUNGSDESIGN (METHODOLOGIE/METHODEN/BASIS)
  → DURCHFUEHRUNG Steps 1-4 (Hotspots + H1 + Grounding + BEFUND-Konsolidierung)
  → SYNTHESE (GESAMTERGEBNIS)
  → SCHLUSSREFLEXION (GELTUNGSANSPRUCH)
```

Reihenfolge ist von Cross-Typ-Bezügen erzwungen (siehe Memory `project_three_heuristics_architecture.md`).

**Fallback-Logik**: wenn vorgelagerter Pass fehlt (z.B. EXPOSITION nicht gelaufen), Container-Orchestrator soll early-error mit klarer Fehler-Message, nicht Kaskaden-Crash. Bestehende Heuristiken werfen das schon — Orchestrator muss nur sammeln.

**Idempotenz**: jede einzelne Heuristik ist idempotent. Re-Run des Orchestrators = jede Heuristik re-läuft = erzeugt gleichen Output (deterministischer LLM-Call modulo Temperature). Optional: `--skip-existing` Flag, das Pässe überspringt, deren Konstrukte schon vorliegen.

**CLI-Test**: `scripts/test-h3-orchestrator.ts <caseId> [--skip-existing] [--from=<step>]`.

**Status-Doku**: `docs/h3_orchestrator_status.md` neu anlegen.

**Bestehende Spec-Datei**: `docs/h3_orchestrator_spec.md` ist im Repo unversioniert (nicht aus dieser Session). Vor Implementation prüfen, was dort drin steht — könnte schon Ideen/Specs haben, die zu integrieren sind.
